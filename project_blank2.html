json web tokens

Met het surfen op het internet komen wij allemaal in aanraking met veel verschillende websites. Voor veel websites en services op het internet hebt u een account nodig en omdat mensen niet graag inloggen wordt deze authenticatie bijgehouden op verschillende manieren. In deze Magnus opus ga ik het hebben over web tokens die daarvoor gebruikt kunnen worden.

Sessies of cookie-based authentication:

Wanneer iemand zich authentiseert wordt er op de server er sessie gemaakt met de informatie van user gekoppeld aan credentials. De credentials worden in cookies doorgestuurd en elke keer dat de user zich moet authentiseren word de credentials van de user op de server opgezocht.

Voordelen:

Omdat de server alle records heeft is het makkelijk om acces weg te nemen en is er veel trust dat de user gegevens juist zijn. Er is minder verkeer over het netwerk want credentials kan 1 string zijn.

Nadelen:

Voor veel users moet de database die de sessies beheerd overall bereikbaar zijn en heel veel request verwerken dit kan voor vertraging zorgen

Json Web tokens:

Json web tokens zijn bijna hetzelfde als sessies met een groot verschil dat de userinformatie niet op de server in een database wordt opgeslagen maar in de string die word gegeven aan de user.Een json web token heeft 3 secties header met informatie over encryptie algoritme dat gebruikt wordt. De payload met informatie bv username, id, tijd dat de token geldig is. De signature met de hash om ervoor to zorgen dat er niks is veranderd met de token.

Voordelen:

Snelheid van authenticatie hoger omdat er geen vraag naar een database is. Schaalbaarheid is ook groter niet gelimiteerd door database

Nadelen:

Wanneer een web token gemaakt is het niet mogelijk om informatie erin te updaten Om out te loggen op elk device is niet praktisch zonder te wachten op de vervaltijd. Je kunt gemakkelijk meerdere servers in verschillende domainen met dezelfde tokens werken
